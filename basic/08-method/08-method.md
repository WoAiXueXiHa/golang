## 🧱 模块一：方法的本质——带特定接收者暗号的普通函数

### 1. 完整代码案例

```go
package main

import "fmt"

type Counter struct {
	Count int
}

// 指针接收者方法
func (c *Counter) AddPtr() {
	c.Count++ // 底层顺着物理地址修改原件
}

// 值接收者方法
func (c Counter) AddValue() {
	c.Count++ // 底层只修改了传入的值副本
}

func main() {
	fmt.Println("----------- 方法基本使用 ----------")
	c := Counter{Count: 1} // 显式初始化值类型变量
	
	// 🎯 编译器自动取地址语法糖：
	// c 是一个值类型，虽然 AddPtr 需要指针，但因为 c 是可取地址的变量，
	// 编译器自动将其转换为 (&c).AddPtr()，从而安全修改真身
	c.AddPtr()
	fmt.Println("指针方法执行后: ", c.Count) // 打印结果: 2

	// 🎯 编译器自动解引用语法糖：
	// 如果用的是指针变量（如通过 new 创建），调用值方法时，
	// 编译器会自动帮我们隐式转换为 (*指针).AddValue()
	c.AddValue()
	fmt.Println("值方法执行后: ", c.Count) // 打印结果: 2 （值方法拷走了副本，不影响真身）
}

```

### 2. 代码逐行解释

* `func (c *Counter) AddPtr()`：在函数名前面增加的小括号 `(c *Counter)` 叫做**接收者（Receiver）**。它在编译期通过静态词法分析后，本质上会沦为这个函数的**第一个隐式形参**。
* `c.AddPtr()`：值类型对象直接调用指针方法。此时 Go 编译器发现当前作用域内的 `c` 是一个完好的局部栈变量，可以安全地通过 `&` 获取地址，于是隐式重写代码，完成调用。

### 3. 底层内存原理 🧠

配合 **【第一部分图1、图2】**：我们在第一章看过的硬核反汇编（`go tool objdump`）在这里形成了绝对的逻辑闭环！
在底层的机器码中，没有所谓的方法，全部都是带有特定包名暗号的普通函数。

* 指针方法 `func (c *Counter) AddPtr()` 编译后现出原形：`main.(*Counter).AddPtr(SB)`。
* 值方法 `func (c Counter) AddValue()` 编译后现出原形：`main.Counter.AddValue(SB)`。
当值变量 `c` 调用指针方法时，CPU 的 `AX` 寄存器里塞进去的是 `c` 实例在栈上的**物理内存地址**。函数内部自增时，是顺着这个物理地址直接敲击 `main` 栈帧里的原始内存。
而调用值方法时，寄存器塞进去的是数据的**值拷贝**。在函数体内自增后，随着函数栈帧退弹，这块临时修改的内存直接在时空中湮灭，外界的原件完全没有被触碰。

## 🧱 模块二：组合代替继承（内存平铺与方法提升）

### 1. 完整代码案例

```go
package main

import "fmt"

type Engine struct {
	Power int
}

func (e *Engine) Start() {
	fmt.Println("引擎启动...")
}

type Car struct {
	Name string
	Engine // 🤝 匿名嵌入（正交组合的精髓）
}

// 覆写（Override）外层同名方法
func (c *Car) Start() {
	fmt.Printf("%s 跑车正在启动...\n", c.Name)
}

func main() {
	fmt.Println("----------- 组合 ----------")
	myCar := Car{
		Name:   "Audi",
		Engine: Engine{Power: 500},
	}
	
	// 🚀 字段提升：可以直接使用 myCar.Power 绕过内层结构体名
	fmt.Println("马力：", myCar.Power) // 打印结果: 500
	
	// 🚀 Override：优先调用外层自己重写的方法
	myCar.Start() // 打印结果: Audi 跑车正在启动...
	
	// 🚀 显式调用：如果想用内部被隐藏的方法，可以通过结构体名精准定位
	myCar.Engine.Start() // 打印结果: 引擎启动...
}

```

### 2. 代码逐行解释

* `type Car struct { Name string; Engine }`：没有给 `Engine` 起变量名，直接把类型名丢了进来。这在 Go 语言中叫做**匿名嵌入（Anonymous Embedding）**，是正交组合的基石。
* `myCar.Power`：字段提升（Field Promotion）。编译器在编译期如果在外层找不到 `Power` 字段，会自动递归去匿名嵌入的内部组件里检索。如果有，就自动帮我们重定向过去。

### 3. 底层内存原理 🧠

配合 **【第二部分图3】**：大厂高级面试时最喜欢考：Go 的组合在内存里是不是像 Java 的继承一样，有复杂的父子对象引用嵌套和虚函数表？
**答案是：完全没有！Go 的组合是极致高效的“内存平铺（Flattened Memory Layout）”！**

我们在创建 `myCar` 时，在内存条上开辟的是一块连续的物理存储空间。前 16 个字节放的是外层的 `Name`（字符串结构体），紧随其后的 8 个字节直接放的就是内层 `Engine` 的 `Power` 字段！
它们物理上紧紧挨在**同一块连续的内存蛋糕**里。所谓字段提升和方法提升，纯粹是 Go 编译器在编译阶段根据内存偏移量（Offset）帮你算好了数据存放的绝对地址，生成的机器码运行效率高到极限，完全没有运行时指针重定向的额外开销！

## 🧱 模块三：方法集（Method Set）铁律与接口锁死机制

### 1. 完整代码案例

```go
package main

import "fmt"

// 定义统一的支付接口标准
type Payer interface {
	Pay(amount int)
}

type WeChatPay struct {
	Balance int
}

// 扣钱业务必须影响真身，所以必须使用指针接收者
func (w *WeChatPay) Pay(amount int) {
	w.Balance -= amount
	fmt.Printf("微信支付成功，扣除 %d 元， 余额 %d 元\n", amount, w.Balance)
}

func main() {
	fmt.Println("----------- 方法集 ----------")
	wxWallet := WeChatPay{Balance: 100}
	
	// ❌ 错误示范（解除原代码中此行的注释会引发编译报错）：
	// var p1 Payer = wxWallet 
	// 🛑 报错信息：cannot use wxWallet (variable of type WeChatPay) as Payer value in variable declaration: missing method Pay
	
	// ✅ 正确示范：传递指针对象
	var p2 Payer = &wxWallet
	p2.Pay(20) // 顺利扣款
}

```

### 2. 代码逐行解释

* `type Payer interface { Pay(amount int) }`：定义一个标准的支付接口。
* `var p1 Payer = wxWallet`：尝试把结构体**值类型变量**赋值给接口。编译器直接在编译阶段无情拦截，宣告失败。
* `var p2 Payer = &wxWallet`：把结构体**指针变量**赋值给接口。由于指针拥有全能型的方法集，编译完美通过。

### 3. 底层内存原理 🧠

配合 **【第三部分图4、图5】**：为什么 Go 编译器要这么冷酷？为什么普通值变量 `wxWallet` 明明可以取地址，却“没有资格”冒充 `Payer` 接口？

* **大厂终极面试内幕**：还记得我们之前学过的接口底层结构体 **`iface`** 吗？接口在赋值时，由于底层发生值拷贝，会将具体的数据复制一份放进 `iface.data` 指针指向的内存中。
* 如果你允许 `var p1 Payer = wxWallet`，那么 `p1.data` 里面装的其实是一个全新的、隔离出来的 **`wxWallet` 的值副本**。
* 此时如果允许你调用 `p1.Pay(20)`，因为 `Pay` 方法指定了必须用指针接收者 `(w *WeChatPay)`，Go 运行时就需要自动在底层调用时**去获取这个接收者的物理地址**。
* 但是，因为 `p1.data` 内部包裹的是一个**受保护的、不可寻址的、处于黑盒状态的接口副本值**，Go 运行时在底层绝对无法、也不允许去获取它内部的物理地址！如果强行放行，这个指针在修改完副本后就会直接悬空，导致外界真正的原件血量/余额根本没变，引发毁灭性的数据错乱。
* **官方铁律总结：** * **类型 `T`（值类型）的方法集**：只包含值接收者方法。
* **类型 `*T`（指针类型）的方法集**：包含值接收者 + 指针接收者方法（全能选手）。
为了从根本上掐灭一切因为副本不可寻址而导致“指针悬空、修改丢失”的隐患，Go 编译器直接在编译阶段通过方法集铁律，把你危险的试探无情锁死！
