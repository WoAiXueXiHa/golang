# 并发控制的底层原理

## 这一章要记住什么

- **竞态条件**的根源是"读-改-写"不是原子操作——多个线程同时修改同一变量时，修改会被覆盖
- **信号量（PV 操作）** 是并发控制的理论基石：P 申请资源，V 释放资源；容量为 1 就是互斥锁
- **死锁**需要四个条件同时成立：互斥、持有并等待、不可剥夺、循环等待——破坏任一条件就能预防
- **读写锁**把读者和写者分开：读者之间不互斥，写者独占，适合读多写少的场景
- **原子操作**靠 CPU 硬件指令保证不可中断，比锁更快但只适用于简单变量操作；CAS 是无锁编程的核心

---

## 1. 竞态条件：为什么并发会出问题

### `num += 1` 不是一步完成的

这行看起来简单的高层语言代码，在 CPU 层面其实是**三条指令**：

```
高级语言：num += 1

CPU 实际执行：
┌──────────┐    ┌──────────┐    ┌──────────┐
│ 1. LOAD  │ → │ 2. ADD   │ → │ 3. STORE │
│  num → R  │   │  R + 1   │   │  R → num │
└──────────┘    └──────────┘    └──────────┘
从内存读 num    寄存器加 1      写回内存
```

### 两个线程同时执行时发生什么

```
时间线 →

线程 A：LOAD num(0) ──→ ADD(1) ──→ STORE num(1) ───────────
                                      │
线程 B：LOAD num(0) ──→ ADD(1) ──→ STORE num(1)   ← 覆盖！
              ↑
        A 还没来得及写回，B 读到的是 0

期望结果：num = 2（加了两次）
实际结果：num = 1（A 的修改被 B 覆盖）
```

这就是**竞态条件（Race Condition）**：多个线程并发访问同一数据，最终结果取决于线程执行的**相对时序**——时序是无法预测的，所以程序时而正确时而出错。

### 临界区

被多个线程访问、且至少有一个线程在修改的代码段，叫做**临界区（Critical Section）**。并发编程的核心问题就是：**如何保证同一时刻最多只有一个线程在临界区里**。

### 总结一下

一行 `num += 1` 在 CPU 层面是读-改-写三条指令，多核上它们交叉执行导致修改互相覆盖。并发安全的本质是控制"同时进入临界区"的线程数量——有且仅有一个。

## 2. 信号量与 PV 操作

1965 年，荷兰计算机科学家 Dijkstra 提出了**信号量（Semaphore）** 这个概念，它是所有并发控制原语的理论祖先。

### 信号量的直觉模型

想象一个公共厕所，只有 **N 个坑位**：

```
信号量 s = 坑位数（初始值 N）

一个人想进去：
  P(s)：坑位 - 1，如果坑位已经为 0，就在门口排队
  
一个人出来：
  V(s)：坑位 + 1，如果有排队的人，唤醒一个进去
```

### PV 操作的精确定义

```
P(s)：申请资源（荷兰语 Proberen = "尝试 / 检测"）
  如果 s > 0：
    s = s - 1   → 拿到资源，进去
  如果 s == 0：
    把当前线程挂起，加入 s 的等待队列   → 没有资源，原地等

V(s)：释放资源（荷兰语 Verhogen = "增加"）
  如果有人在等 s：
    唤醒等待队列中的第一个线程  → 把资源给等的人
  如果没有人在等：
    s = s + 1   → 把资源还回信号量
```

**关键：P 和 V 操作本身必须是原子的**——CPU 不能在做 P 操作的中途被另一个线程打断。这个原子性由 OS 或硬件保证（通常会配合关中断或原子指令）。

### 信号量的两个用途

**用途一：互斥锁（容量 s = 1）**

```
s = 1（toilet 只有一个坑位）

线程 A: P(s) → s = 0 → 进入临界区
线程 B: P(s) → s = 0，当前 == 0 → 堵在门口排队
线程 A: V(s) → 发现 B 在等 → 唤醒 B

同一时刻只有一个线程在临界区里 ✓
```

**用途二：同步（容量 s = 0）**

```
s = 0（初始没有资源）

线程 A: 必须先有 V 才能继续 → 在 P(s) 等待
线程 B: 执行完某段逻辑 → V(s) → 唤醒 A

A 必须等 B 完成某个任务之后才能继续，这就是"同步"的含义
```

### 总结一下

信号量是一个整数计数器 + 等待队列。P 操作相当于"申请一个资源"，资源不够就排队等；V 操作相当于"归还一个资源"并唤醒等待者。容量设为 1 就是互斥锁，容量设为 0 就是同步通知机制。Go 的 channel 本质上就是一个带了类型系统的信号量。

## 3. 互斥锁（Mutex）

**Mutex = "Mutual Exclusion"（互斥）**，本质上就是容量为 1 的信号量——二元信号量。

### 锁是怎么阻塞和唤醒线程的

```
goroutine A 想获取锁：
    
Lock() ──→ 检查锁的状态
              │
              ├── 锁空闲 → 标记为已占用，立刻返回
              │
              └── 锁被占用 → 调用 OS 调度器
                               把你从"运行队列"移到"等待队列"
                               OS 不再给你分配 CPU 时间
              
goroutine B 释放锁：
    
Unlock() ──→ 标记锁为空闲
              │
              └── 如果有 goroutine 在等待队列
                   把第一个从等待队列移到运行队列
                   OS 调度器下次会分配 CPU 给它
```

```
运行队列（可以获得 CPU）              等待队列（不能获得 CPU）
┌──┬──┬──┬──┐                     ┌──┬──┬──┐
│G1│G3│G4│G5│                     │G2│G6│..│
└──┴──┴──┴──┘                     └──┴──┴──┘
      ▲                                  │
      │  G2 获锁失败                    │  G0 Unlock 后唤醒 G2
      └──────────────────────────────────┘
```

**自旋 vs 睡眠：**
- **睡眠锁**（Go 的 Mutex）：等不到就让出 CPU，去睡觉。适合锁持有时间长的场景。
- **自旋锁**：等不到就原地循环检查，不让出 CPU。适合锁持有时间极短（几条指令就释放），因为睡觉再被唤醒的上下文切换开销可能比自旋等还大。

Go 1.18+ 的 Mutex 实际上两种都用了——先自旋等一小会儿，自旋几次还没抢到就切到睡眠。

### 总结一下

互斥锁是容量为 1 的信号量，保证同一时刻只有一个 goroutine 在临界区。等不到锁的 goroutine 被 OS 从运行队列移走，释放锁时再唤醒。Go 的 Mutex 先自旋再睡眠，兼顾了短等待和长等待的效率。

## 4. 读写锁

### 为什么需要读写锁

很多场景下，**读操作远多于写操作**。比如配置信息，一小时才改一次，但每秒被读几万次。如果用互斥锁，所有读操作也要互斥——同一时刻只能有一个人在读，白白浪费并发能力。

**读操作本身不修改数据，多个读者完全可以同时读，谁也不会干扰谁。**

这就是读写锁的逻辑：

```
读锁（共享锁）：
  多个 goroutine 可以同时持有
  只要没有人在写，你就可以读

写锁（排他锁）：
  只有一个 goroutine 能持有
  写锁和读锁、写锁和写锁都互斥
```

### 四个规则

```
┌──────┬──────┬──────┐
│      │ 读锁 │ 写锁 │
├──────┼──────┼──────┤
│ 读锁 │ ✅  │ ❌   │   读读共享
├──────┼──────┼──────┤
│ 写锁 │ ❌   │ ❌   │   读写互斥、写写互斥
└──────┴──────┴──────┘
```

```
当前状态：没有锁

Reader A: RLock() → ✅ 拿到读锁，cnt = 读锁计数 1
Reader B: RLock() → ✅ 拿到读锁，cnt = 读锁计数 2
Reader C: RLock() → ✅ 拿到读锁，cnt = 读锁计数 3

Writer D: Lock()  → ❌ 阻塞！有读者在，写者必须等所有读者释放
                     
Reader A: RUnlock() → cnt = 2
Reader B: RUnlock() → cnt = 1
Reader C: RUnlock() → cnt = 0  → 所有读者释放 → 唤醒写者 D

Writer D: 获得写锁 → 写操作 → Unlock()
```

### 读多写少的性能收益

假设有 100 个读操作和 2 个写操作，每个耗时 1ms：

| 方案 | 读耗时 | 写耗时 | 总耗时 |
|------|--------|--------|--------|
| 互斥锁 | 100ms（排队串行） | 2ms | ~102ms |
| 读写锁 | ~1ms（全部并发读） | 2ms | ~3ms |

读写锁在这种场景下收益非常明显。

### 总结一下

读写锁把读者和写者分开对待：读者之间不互斥（共享锁），写者独占（排他锁）。读多写少的场景能大幅提升并发性能。但要记住：只要有一个写者，它就要和所有读者互斥。

## 5. 死锁

### 什么是死锁

两个或更多 goroutine 互相持有对方需要的资源，谁也不肯放手，大家一起无限等下去。

### 循环等待是最经典的死锁场景

```
goroutine A                          goroutine B
───────────                          ───────────
Lock(mu1) ✅                         Lock(mu2) ✅
如获至宝，继续执行...                 如获至宝，继续执行...

Lock(mu2) ⏸                          Lock(mu1) ⏸
等 B 释放 mu2...                     等 A 释放 mu1...

         A 等 B，B 等 A → 死锁 🔒
```

```
        ┌─────────→ mu1 ←──────────┐
        │  持有                    │  等待
        │                          │
     goroutine A               goroutine B
        │                          │
        │  等待                    │  持有
        └─────────→ mu2 ←──────────┘

           形成闭环 → 谁也无法前进
```

### 死锁的四个必要条件（Coffman 条件）

死锁发生必须**同时**满足这四个条件，缺一个就不会死锁：

| # | 条件 | 含义 | 类比 |
|---|------|------|------|
| 1 | **互斥** | 资源不能同时被多个 goroutine 使用 | 一把锁同时只能一个人拿 |
| 2 | **持有并等待** | 拿着已有的资源，等新的资源 | 拿着 mu1，等 mu2 |
| 3 | **不可剥夺** | 别人不能强行抢走你的资源 | goroutine 不 Unlock，谁也拿不走 |
| 4 | **循环等待** | 形成 A→B→C→A 的等待环 | goroutine A 等 B，B 等 A |

### 怎么预防死锁

破坏任何一个条件就行：

1. **破坏互斥**——把共享资源改成不共享的。比如每个 goroutine 用自己的局部变量，通过 channel 通信（Go 的哲学）。
2. **破坏持有并等待**——一次性申请所有需要的锁，拿不到就全部释放。
3. **破坏不可剥夺**——如果拿不到第二个锁，就把第一个锁也放掉，重新抢。`sync.Locker` 没有这个能力，但可以用 `tryLock`（Go 1.18+）或者 channel 来做超时。
4. **破坏循环等待**——**规定锁的获取顺序**。所有 goroutine 都按相同的顺序加锁（比如永远先 Lock(mu1)，再 Lock(mu2)），就不会形成环。

### 总结一下

死锁需要四个条件同时成立：互斥、持有并等待、不可剥夺、循环等待。预防死锁的关键是破坏任一条件——最常用的方法是规定所有 goroutine 按相同顺序加锁，避免形成循环等待环。

## 6. 原子操作

### 什么是原子操作

原子操作就是**CPU 硬件层面保证不会被中断的操作**。一条 CPU 指令就能完成，中间不可能被其他核上的线程打断。

和互斥锁的关键区别：

```
Mutex 保护临界区                原子操作
─────────────────              ─────────
临界区里可以有多条指令          一条 CPU 指令搞定
涉及 OS 调度器                  不涉及 OS 调度器
等不到就要上下文切换            不会阻塞，立即执行完毕
重量级（几百个 CPU 周期起）     轻量级（几个 CPU 周期）
保护任意代码                    只能操作单个变量
```

### CAS（Compare And Swap）

CAS 是原子操作的核心，几乎所有无锁数据结构的基础：

```go
// 伪代码：CAS 是一条 CPU 指令，不可被中断
func CompareAndSwap(addr *int32, old, new int32) bool {
    if *addr == old {
        *addr = new   // 如果 addr 的当前值等于 old，就换成 new
        return true
    }
    return false      // 不等于，说明别人已经改了，本次失败
}
```

CAS 的使用模式：

```
1. 读出当前值：old = Load(addr)
2. 算出新值：   new = old + 1
3. CAS(addr, old, new)：如果 addr 没被别人改过（还等于 old），就写入 new
4. 如果 CAS 返回 false（别人抢先改了），回到步骤 1 重试
```

这就是**无锁编程**的基本思路：不用锁，而是用 CAS 反复尝试，直到成功。

### 原子操作的局限性

原子操作很快，但它只能保护**单个变量的简单操作**——加一、赋值、交换。如果要跨多个变量做事务性修改（比如转账：A 余额 -100 同时 B 余额 +100），原子操作就不够用了，必须上锁。

### 总结一下

原子操作靠 CPU 硬件指令保证不可中断，快且不阻塞，是 Mutex 在简单场景下的高性能替代方案。CAS 是原子操作的核心——"确认值没变，才写入"，变了就重试。适合单个变量的操作，复杂逻辑还是要用锁。

## 快问快答

### Q1：`num += 1` 为什么不是线程安全的？

答：因为它在 CPU 层面是三条指令——LOAD、ADD、STORE。多核 CPU 上两个线程可能同时 LOAD 到同一个值，各自加 1 后写回，其中一个结果被覆盖。最终 num 只加了 1 而不是 2。

### Q2：信号量的 P 和 V 操作分别做什么？

答：P（申请）——如果信号量值大于 0，就减 1 拿到资源；等于 0 就把当前线程挂起等资源。V（释放）——如果有线程在等，就唤醒等待队列里的第一个；没人在等就把信号量值加 1。P 和 V 本身必须是原子的。

### Q3：互斥锁和读写锁的区别？

答：互斥锁只允许一个线程持有锁，不管读还是写都是独占。读写锁把读者和写者分开——允许多个读者同时持有读锁，但写锁和读锁、写锁之间都是互斥的。适合读多写少的场景，读并发能力比互斥锁高很多。

### Q4：死锁的四个条件是什么？

答：互斥（资源不能共享）、持有并等待（拿着一个等另一个）、不可剥夺（只能自己放手，不能被抢走）、循环等待（A 等 B 等 C 等 A 形成环）。四个条件缺一不可，破坏任何一个就能预防死锁。

### Q5：原子操作和锁有什么区别？

答：原子操作是 CPU 指令级别的，不涉及 OS 调度，不阻塞，非常快——适合单个变量的简单操作（计数、标志位切换）。锁是 OS 调度器管理的，等不到就让出 CPU，可以保护任意代码块。原子操作是"轻量级办法"，锁是"万能办法"。

### Q6：CAS 是什么，怎么用它实现无锁更新？

答：CAS（Compare And Swap）是一条 CPU 原子指令，比较地址的当前值和期望值，相同就写入新值，不同就放弃。用在循环里：读出值 → 算出新值 → CAS 尝试写入 → 失败（被抢先了）就重读重试。这就是无锁编程的核心手段。

---

## 一句话总结

并发安全的核心是控制多个执行流同时访问共享数据：信号量提供了理论基础（PV 操作），互斥锁是最常用的保护手段（二元信号量），读写锁在"读多写少"场景下更有优势，死锁要防四个条件同时成立，原子操作是 CPU 指令级的轻量方案适合简单变量场景——理解这五样，并发编程的底层逻辑就通了。
