## 🧱 模块一：前置操作系统底蕴与主协程偷跑宿命

### 1. 完整代码案例

```go
package main

import (
	"fmt"
	"time"
)

func myGoroutine() {
	fmt.Println("myGoroutine")
}

func main() {
	fmt.Println("----------- 1. 前置知识与主协程生命周期 ----------")
	
	// 启动子协程
	go myGoroutine()
	
	fmt.Println("end--------")
	
	// 🎯 悬学点破：如果不加下面这一行，控制台将【绝对看不到】"myGoroutine" 的打印！
	// 因为 main 函数是默认的主协程，它一旦执行完最后一行机器码，会直接通知
	// 操作系统销毁整个进程，根本不会等还在后台排队等待调度的子协程！
	time.Sleep(2 * time.Second) // 靠肉眼盲目猜测并休眠 2 秒
}

```

### 2. 代码逐行解释

* `go myGoroutine()`：`go` 是 Go 语言唯一的并发关键字。它告诉 Go 运行时（Runtime）：“请立刻在堆上为 `myGoroutine` 划拨一块 2KB 的虚拟栈空间，打包成一个 G 结构体，丢进运行队列等待调度”。
* `time.Sleep(...)`：由于主协程（`main` 函数）执行完就会直接结束进程，导致子协程直接被掐死在摇篮里。这里用休眠 2 秒来强行续命。大厂工业级开发中**绝对禁止使用 Sleep 来等待协程**，必须使用你在下一个模块中写到的 `sync.WaitGroup`。

### 3. 底层内存原理 🧠

配合 **【ASCII 全景图第一部分】**：

* **进程**：是操作系统分配资源的最小单位。它拥有独立的虚拟内存、堆空间、进程控制块（PCB）。
* **线程**：是 CPU 调度的最小单位。传统的内核线程非常重（通常占 1MB~8MB 的栈内存），线程间的轮换（上下文切换）需要陷入内核态，开销极大。
* **协程（Goroutine）**：是用户态的轻量级执行流。它的初始栈只有极其轻量的 **2KB**，且完全由 Go 语言的运行时调度器（GMP 模型）在用户态进行管理。上万个协程的上下文切换仅仅是几个 CPU 寄存器的值拷贝，因此 Go 能够轻松支撑百万级的并发。

## 🧱 模块二：多协程优雅同步等待（sync.WaitGroup 财务账本）

### 1. 完整代码案例

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func worker(id int, wg *sync.WaitGroup) {
	// 🎯 破局铁律：必须用 defer 将 Done 锁死在第一行！
	// 这样无论后续业务正常结束还是中途发生 panic，总账本都能得到扣减，绝育死锁。
	defer wg.Done()

	fmt.Printf("worker[%d] 开始工作。。。\n", id)
	time.Sleep(time.Second * 1)
	fmt.Printf("worker[%d]结束工作。。。\n", id)
}

func main() {
	fmt.Println("----------- 2. 多协程调用和等待 ----------")
	var wg sync.WaitGroup

	for i := 1; i <= 3; i++ {
		// 🎯 记账防线：必须写 1，循环 3 次，总账本记 3 笔。
		// 如果写成 wg.Add(2)，主协程就会因为算错总账、永久等不到归零而发生 Deadlock 崩溃！
		wg.Add(1)

		go worker(i, &wg) // 必须传指针 &wg，防止值拷贝导致锁失效
	}

	fmt.Println("------------ 主协程开始等待 ----------")
	
	// 🎯 阻塞点：主协程让出 CPU，陷入休眠。直到账本扣减为 0 时被精准唤醒
	wg.Wait()

	fmt.Println("-------- 主协程退出 -------------")
}

```

### 2. 代码逐行解释

* `wg.Add(1)`：原子计数器加 1。代表派出去一个并发任务。必须写在 `go` 启动之前，防止主协程跑太快直接冲到 `Wait()`。
* `go worker(i, &wg)`：**指针传递的经典运用**。我们在第二章学过，Go 只有值传递。如果不加 `&`，会克隆出一把全新的等待组锁，子协程在副本上扣减，主协程的账本永远无法归零，导致程序永久死锁。

## 🧱 模块三：多协程异常捕获与工业级防御外壳（SafeGo）

### 1. 完整代码案例

```go
package main

import (
	"fmt"
	"sync"
	"time"
)

func badCapture() {
	// ❌ 错误示范：企图在主协程的栈上架设捕获网
	defer func() {
		if err := recover(); err != nil {
			fmt.Printf("试图捕获子协程异常: %v\n", err)
		}
	}()

	go func() {
		fmt.Println("子协程：我要 panic 了")
		panic("子协程 panic ") // 💥 核心铁律：这里会直接绕过外层的网，造成进程自杀！
	}()

	time.Sleep(time.Second * 1)
}

func testCaptureConut() {
	defer func() {
		// 🎯 核心考点：捕获次数判定
		if err1 := recover(); err1 != nil {
			fmt.Printf("第一次执行 recover 捕获成果: %v\n", err1)
		}

		// 紧接着调用第二次：
		if err2 := recover(); err2 != nil {
			fmt.Printf("第二次执行 recover， 成功捕获: %v\n", err2)
		} else {
			fmt.Println("第二次执行 recover， 返回 nil， 证明一个 panic 只能 recover 一次!")
		}
	}()

	panic("触发")
}

// 🛡️ 工业级大厂高并发降级规范：SafeGo 安全外壳
func SafeGo(wg *sync.WaitGroup, task func()) {
	wg.Add(1)
	go func() {
		defer wg.Done()
		
		// 🎯 核心考点：捕获范围闭环。让每个子协程的调用栈顶都自带一张安全网
		defer func() {
			if err := recover(); err != nil {
				// 线上微服务中，这里通常会接入告警系统
				fmt.Printf("拦截成功，检测到子协程 panic， 已经成功隔离，原因为: %v\n", err)
			}
		}()

		// 真正执行业务逻辑
		task()
	}()
}

func main() {
	fmt.Println("------------ 3. 多协程异常捕获和捕获次数测试 ---------------")
	testCaptureConut()

	time.Sleep(time.Second * 1)
	fmt.Println("-------------- 并发防止崩溃测试 ---------------")

	var wg sync.WaitGroup

	SafeGo(&wg, func() {
		fmt.Println("任务一：我是健康的业务，正常执行")
	})

	SafeGo(&wg, func() {
		fmt.Println("任务二：我很危险，我要踩雷了")
		a := 1
		b := 0
		fmt.Println(a / b) // 💥 触发除零崩溃
	})

	wg.Wait()
	fmt.Println("虽然任务二崩溃了，但是因为 SafeGo 的隔离，主进程顺利完成！")

	// badCapture() // 解除此行注释可观摩整个进程毁灭的惨烈现场
}

```

### 2. 代码逐行解释与底层原理 🧠

配合 **【ASCII 全景图第二部分】**。

* `testCaptureConut` 中的第二次 `recover()` 返回 `nil`：我们在第五章学过，`recover` 的本质是去消除当前协程结构体里的崩溃标志。一旦第一次执行成功，标志位就会被彻底擦除，Goroutine 回归正常状态，因此第二次调用必定返回 `nil`。**一个 Panic 只能被消费一次。**
* `badCapture()` 为什么注定失败？因为 `go func()` 开启的是一个**物理上完全隔绝的全新调用栈**。Go 运行时的底层 `g` 结构体中，`_panic` 和 `_defer` 链表是**协程之间完全独立、互不相通的**。当子协程发生崩溃时，运行时只会在它自己的调用栈上逆向寻找 `defer`。因为 `badCapture` 的 `recover` 写在主协程的栈里，子协程根本摸不到，崩溃流最终无路可走，触发 `dieFromPanic` 信号，拉着整个主进程一起玉石俱焚。
* `SafeGo` 为什么能拯救线上？因为它把 `defer recover()` 牢牢绑定在了**子协程自己的栈最底部**。崩溃流在冲出子协程、毁灭进程之前，必然会被自己栈底的 `recover` 拦截吞噬。这在架构设计上实现了**粉碎性崩溃的完美安全隔离**。
